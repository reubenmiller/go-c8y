# Cumulocity Go SDK v2

> **See [API_DESIGN.md](./API_DESIGN.md) for comprehensive API design documentation.**

This v2 rewrite of the go-c8y SDK provides a modern, type-safe, and flexible Go client for Cumulocity IoT. The design emphasizes developer experience, type safety through generics, and rich result metadata.

## Key Features

### Rich Result Type
Every operation returns a `Result<T>` with comprehensive metadata including status, HTTP details, retry information, duration, and request inspection capabilities.

### Flexible JSON Handling
JSONDoc models provide both structured method access and raw JSON access, allowing custom fields and unmarshaling to user-defined types.

### Context-Driven Behavior
Control execution through context:
- **Dry Run** - Prepare requests without sending
- **Deferred Execution** - Resolve parameters, then execute after confirmation
- **Header Redaction** - Control sensitive data visibility in logs

### Smart Resource Resolution
Consistent resolver patterns across services:
- Direct IDs: `"12345"`
- Name lookup: `"name:deviceName"`
- External IDs: `"ext:c8y_Serial:ABC123"`
- Query-based: `"query:type eq 'c8y_Device'"`

### Powerful Pagination
- Collection results with metadata
- Iterator pattern for all items
- Custom unmarshaling during iteration

## Quick Example

```go
package main

import (
    "context"
    "fmt"
    "github.com/reubenmiller/go-c8y/pkg/c8y/c8y_api"
    "github.com/reubenmiller/go-c8y/pkg/c8y/c8y_api/operations"
)

func main() {
    // Create client
    client := c8y_api.NewClient(baseURL, username, password)
    ctx := context.Background()
    
    // List operations with device name resolution
    result := client.Operations.List(ctx, operations.ListOptions{
        DeviceID: "name:my-device",  // Automatically resolved to ID
        Status:   "PENDING",
    })
    
    // Handle result
    if result.Err != nil {
        panic(result.Err)
    }
    
    // Iterate over operations
    for _, op := range result.Data.Items() {
        fmt.Printf("%s: %s\n", op.ID(), op.Status())
    }
}
```

## Pipeline & Streaming Workflows

The `pipeline` package provides concurrent iteration utilities inspired by go-c8y-cli's pipe-based command chaining. It builds on Go 1.23's `iter.Seq[T]` — the same type returned by `ListAll(...).Items()` — so any paginated query, slice, or channel can be processed with concurrent workers, error control, and progress tracking.

The core API is two functions: `ForEach` (side effects) and `Collect` (gather results).

### Basic: Update Devices (go-c8y-cli equivalent)

```sh
# go-c8y-cli:
c8y devices list --name "MQTT*" -p 1 |
    c8y devices update --template "{name: std.strReplace(input.value.name, 'MQTT Device ', '')}"
```

```go
// Go SDK equivalent:
devices := client.ManagedObjects.ListAll(ctx, managedobjects.ListOptions{
    Query: "name eq 'MQTT*'",
})

err := pipeline.ForEach(ctx, devices.Items(), pipeline.Options{Workers: 5},
    func(ctx context.Context, device jsonmodels.ManagedObject) error {
        newName := strings.TrimPrefix(device.Name(), "MQTT Device ")
        result := client.ManagedObjects.Update(ctx, device.ID(), map[string]any{
            "name": newName,
        })
        return result.Err
    },
)
```

### Cancel Stale Operations

```sh
# go-c8y-cli:
c8y operations list --status PENDING --dateTo "-14d" --includeAll |
    c8y operations update --status FAILED \
        --failureReason "Cancelled stale operation" --workers 3 --delay 250
```

```go
ops := client.Operations.ListAll(ctx, operations.ListOptions{
    Status: "PENDING",
    DateTo: time.Now().AddDate(0, 0, -14),
})

err := pipeline.ForEach(ctx, ops.Items(), pipeline.Options{
    Workers: 3,
    Delay:   250 * time.Millisecond,
},
    func(ctx context.Context, op jsonmodels.Operation) error {
        result := client.Operations.Update(ctx, op.ID(), map[string]any{
            "status":        "FAILED",
            "failureReason": "Cancelled stale operation (older than 14 days)",
        })
        return result.Err
    },
)
```

### Collect Results

Use `Collect` when you need the output of each operation:

```go
names := []string{"sensor-01", "sensor-02", "sensor-03"}

ids, err := pipeline.Collect(ctx, slices.Values(names), pipeline.Options{Workers: 2},
    func(ctx context.Context, name string) (string, error) {
        result := client.ManagedObjects.Create(ctx, map[string]any{
            "name": name,
            "type": "c8y_Sensor",
        })
        if result.Err != nil {
            return "", result.Err
        }
        return result.Data.ID(), nil
    },
)
// ids = ["1234", "5678", "9012"]
```

### Multi-Step Pipelines

Instead of fluent `.Then()` chains, multi-step workflows are just normal Go code inside the callback:

```go
// Create device + register external identity for each
err := pipeline.ForEach(ctx, slices.Values(names), pipeline.Options{Workers: 2},
    func(ctx context.Context, name string) error {
        // Step 1: Create device
        device := client.ManagedObjects.Create(ctx, map[string]any{
            "name": name,
            "type": "c8y_Sensor",
        })
        if device.Err != nil {
            return device.Err
        }

        // Step 2: Create external identity
        identity := client.Identity.Create(ctx, device.Data.ID(), "c8y_Serial", name)
        return identity.Err
    },
)
```

### Error Control

```go
err := pipeline.ForEach(ctx, items, pipeline.Options{
    Workers:        5,
    MaxErrors:      50,   // Stop after 50 failures
    ErrorThreshold: 0.1,  // Stop if >10% fail
},
    func(ctx context.Context, item T) error { ... },
)

// err is *pipeline.AbortError when limits are exceeded
var abortErr *pipeline.AbortError
if errors.As(err, &abortErr) {
    log.Printf("Aborted: %s (failed %d/%d)", abortErr.Reason, abortErr.Failed, abortErr.Completed)
}
```

### Progress Tracking

```go
err := pipeline.ForEach(ctx, items, pipeline.Options{
    Workers: 5,
    OnProgress: func(stats pipeline.Stats) {
        fmt.Printf("Progress: %d completed, %d failed, %d in-flight\n",
            stats.Completed, stats.Failed, stats.InFlight)
    },
},
    func(ctx context.Context, item T) error { ... },
)
```

### Composing with op.Retry and Other Middleware

The existing `op.Step[T]` middleware (Retry, CircuitBreaker, Pipe) composes naturally:

```go
updateWithRetry := op.Retry(func(ctx context.Context, o jsonmodels.Operation) (jsonmodels.Operation, error) {
    result := client.Operations.Update(ctx, o.ID(), body)
    return result.Data, result.Err
}, op.DefaultRetryConfig)

err := pipeline.ForEach(ctx, ops.Items(), pipeline.Options{Workers: 3},
    func(ctx context.Context, item jsonmodels.Operation) error {
        _, err := updateWithRetry(ctx, item)
        return err
    },
)
```

### Integration with Context Features

```go
ctx := context.Background()
ctx = c8ycontext.WithProcessingModeTransient(ctx)            // Transient processing
ctx, cancel := context.WithTimeout(ctx, 10*time.Minute)      // Timeout
defer cancel()

// Pipeline respects all context settings — processing mode, dry run, cancellation
err := pipeline.ForEach(ctx, devices.Items(), pipeline.Options{Workers: 5}, updateFn)
```

## Migration from v1

Key differences:
1. **Result type** instead of `(data, *Response, error)` tuples
2. **Service names** are now plural (`Operations` not `Operation`)
3. **Options structs** instead of variadic functional parameters
4. **JSONDoc models** with method accessors instead of structs

See [API_DESIGN.md](./API_DESIGN.md#migration-from-v1) for detailed migration guide.
